# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

xzSSH is a CLI/TUI SSH-config manager written in Python (3.9+). Source-of-truth lives in a JSON file (`~/.ssh/xzssh.json`); the canonical `~/.ssh/config` is regenerated deterministically. Runtime deps are only `rich` and `questionary`.

## Common commands

Setup (creates `venv/`, installs in editable mode):

```bash
./install.sh           # macOS/Linux
.\install.bat          # Windows
```

After install, the entry point `xzssh` (defined as `xzssh.cli.main:main`) is on `venv/bin/`. Running `xzssh` with no args drops into the interactive welcome menu; subcommands (`list`, `connect`, `add`, `remove`, `import`, `check`, `generate`, `key {add,list,add-agent}`) are also available for scripting.

Tests (install with `pip install -e ".[dev]"` first to get pytest):

```bash
pytest                              # all
pytest tests/test_validator.py      # one file
pytest -k duplicate_alias           # by keyword
```

CI runs the same `pytest` matrix across Linux/macOS/Windows × Python 3.9–3.12 on every push and PR — see [.github/workflows/test.yml](.github/workflows/test.yml).

Release binaries are built by [.github/workflows/release.yml](.github/workflows/release.yml) on `v*` tags: Nuitka compiles `xzssh/cli/main.py` into one-file binaries for Linux / Windows / macOS and attaches them to a GitHub Release. There is no local build target — to test packaging, push a tag or run the workflow via `workflow_dispatch`.

## Architecture

The pipeline is **Parse → Validate → Generate**, and every CLI command flows through it. Each stage is a separate package so the layers stay decoupled; changes that cross layers usually indicate a leak.

- **[xzssh/model/types.py](xzssh/model/types.py)** — Pure dataclasses (`Config`, `Host`, `LocalForward`) with `to_dict()` for JSON serialization. No I/O, no validation. Adding a field means touching this file, the parser, the validator (if it has constraints), and the generator (if it emits to OpenSSH).
- **[xzssh/parser/json_parser.py](xzssh/parser/json_parser.py)** — `load_config(path)` reads JSON, raises `ConfigParseError` on shape errors. Performs only type coercion and required-field checks. Semantic checks belong in the validator. Files with an older schema `version` are upgraded **in memory** through the chain in [xzssh/parser/migrations.py](xzssh/parser/migrations.py) (`CURRENT_SCHEMA_VERSION` lives in the model; the migration contract is documented in the migrations module). The one-time write-back with `.bak` happens in `load_config_if_exists` so it flows through `write_config`. Newer-version files are refused with a parse error.
- **[xzssh/parser/openssh_parser.py](xzssh/parser/openssh_parser.py)** — `parse_openssh_config(path) -> (hosts, warnings)`. Best-effort parser used by `xzssh import`; handles multi-host lines, `=` separators, quoted values, and surfaces `Match`/`Include`/wildcard patterns as warnings rather than following them. No third-party dep — `paramiko.SSHConfig` was deliberately avoided to keep the dep tree small.
- **[xzssh/validator/validate.py](xzssh/validator/validate.py)** — `validate_config(config, suggest_ports, source_path) -> ValidationResult`. Two-bucket result (`errors`, `warnings`). Detects duplicate aliases, port-range issues, and **LocalForward port conflicts across all hosts** (the suggest-ports flag tries to recommend the next free port). Also resolves and stats key files (existence + 0600 perms via `check_private_key_permissions`).
- **[xzssh/generator/openssh.py](xzssh/generator/openssh.py)** — `render_openssh(config, source_path)` returns text. Output is deterministic: a `# Generated by xzSSH - DO NOT EDIT` header followed by hosts in declaration order. Identity files are resolved to absolute paths relative to `source_path.parent`.
- **[xzssh/platform/paths.py](xzssh/platform/paths.py)** — Cross-platform path resolution and POSIX permission enforcement (`chmod 600` on the generated config, mode bits on private keys). Windows path is granted ACL-only with a warning. **All path resolution must go through `resolve_path()`** so `~`, env vars, and source-relative paths behave consistently.
- **`xzssh/cli/`** — thin per-command split:
  - [main.py](xzssh/cli/main.py) — dispatcher only; takes `argv`, builds the parser, calls into one `commands/*` module
  - [parser.py](xzssh/cli/parser.py) — `build_parser()` (argparse setup for every subcommand)
  - [helpers.py](xzssh/cli/helpers.py) — shared helpers: `load_config_or_error`, `load_config_if_exists`, `write_config`, `parse_local_forward_arg`, `resolve_key_path`. **All config writes must go through `write_config`** so JSON formatting stays consistent.
  - [profiles.py](xzssh/cli/profiles.py) — the profile registry (named pointers to alternate config files; JSON at `~/.config/xzssh/profiles.json`, `$XZSSH_PROFILES_FILE` overrides) and `resolve_config_path(config_arg, profile_arg)`, which `main` calls to pick the active config (`--config` > `--profile` > `$XZSSH_PROFILE` > default profile > platform default). The `profile` subcommand is dispatched **before** resolution so a dangling default can't lock the user out of repairing the registry. The test suite pins `XZSSH_PROFILES_FILE` to a tmp path in [tests/conftest.py](tests/conftest.py).
  - [commands/](xzssh/cli/commands/) — one module per subcommand, each exposing a `run(...)`. Leaf commands (`add`, `remove`, `connect`, `import_`, `check`, `generate`, `key`) only depend on `helpers` and `ui`. `list_` also calls into `add`/`remove` for its interactive submenu. `menu` ties everything together — both `default_menu` (no-args welcome) and `main_menu` (`xzssh menu`) live here.
- **[xzssh/cli/ui.py](xzssh/cli/ui.py)** — `rich.Console` themed with neon colors (`#39FF14` green, `#00FFFF` cyan, `#FF00FF` pink) plus `questionary` prompt helpers. UI styling is centralized here — don't sprinkle ANSI or rich markup into command handlers.

### Conventions worth knowing

- **Source paths are anchored.** Relative `identity_file` values resolve relative to the *config file's* directory, not the cwd. This is how `resolve_identity_file` (generator) and `_validate_keys` (validator) both behave — keep this invariant when adding path-bearing fields.
- **The validator is the only place semantic errors are produced.** CLI commands print but don't decide; the parser raises only on JSON shape problems.
- **Writes go through `write_config`** in [xzssh/cli/helpers.py](xzssh/cli/helpers.py), which always serializes `config.to_dict()` with `indent=2`. Don't hand-roll JSON dumps elsewhere.
- **Interactive vs. argument-driven** `add`/`remove` share one handler: missing required args trigger questionary prompts (see [xzssh/cli/commands/add.py](xzssh/cli/commands/add.py)). When extending, keep both the flag form and the prompt form working.
- **`connect` propagates SSH's return code** as the CLI exit code. It only stamps `last_used` when the connection succeeded — i.e. when `ssh` returned anything other than 255 (OpenSSH's convention for "connection setup failed"). Preserve that semantic if you touch [xzssh/cli/commands/connect.py](xzssh/cli/commands/connect.py).
- **Default paths** come from `platform_default_config_path()` / `platform_default_output_path()` (`~/.ssh/xzssh.json` and `~/.ssh/config`). The `--config` and `--output` flags override; never hardcode either path in commands.
